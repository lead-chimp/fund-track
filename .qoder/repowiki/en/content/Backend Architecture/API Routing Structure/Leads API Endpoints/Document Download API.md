## Introduction
This document provides comprehensive documentation for the secure document download endpoint in the fund-track application. The API enables authorized users to retrieve specific documents associated with leads through a secure, time-limited access mechanism. The implementation uses signed URLs to prevent direct exposure of storage paths and ensures proper access control based on user permissions.

## Endpoint Overview
The document download endpoint provides secure access to documents associated with specific leads using nested dynamic parameters.

**Endpoint**: `GET /api/leads/[id]/documents/[documentId]/download`  
**Method**: GET  
**Purpose**: Retrieve a specific document belonging to a lead with secure access control  
**Security**: Requires authentication and authorization validation

The endpoint uses two dynamic parameters:
- `[id]`: The lead identifier
- `[documentId]`: The document identifier within the lead's document collection

## Authentication and Authorization
The endpoint implements a two-step security verification process to ensure only authorized users can access documents.

### Authentication
The endpoint first verifies the user's authentication status using NextAuth.js session management:

```typescript
const session = await getServerSession(authOptions);
if (!session) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}
```

### Authorization
After successful authentication, the endpoint validates that the requested document belongs to the specified lead by querying the database:

```typescript
const document = await prisma.document.findFirst({
  where: {
    id: docId,
    leadId: leadId,
  },
});
```

This query ensures that users can only access documents that are explicitly associated with the lead they are requesting. If no document is found matching both the document ID and lead ID, the endpoint returns a 404 response.

The authorization model follows the principle of least privilege, where access is granted only when both the document exists and belongs to the specified lead.

## Secure URL Generation
The document download endpoint implements time-limited signed URLs to provide secure access to documents without exposing direct storage paths.

### Implementation Details
The endpoint uses the FileUploadService to generate secure download URLs:

```typescript
const downloadResult = await fileUploadService.getDownloadUrl(
  document.b2FileId,
  document.filename,
  1 // 1 hour expiration
);
```

The `getDownloadUrl` method in FileUploadService creates a signed URL with the following parameters:
- **Bucket ID**: The Backblaze B2 bucket identifier
- **File name prefix**: The document's filename
- **Valid duration**: 1 hour (3,600 seconds)

### URL Structure
The generated download URL follows this pattern:
```
https://f005.backblazeb2.com/file/{bucketName}/{fileName}?Authorization={authorizationToken}
```

The authorization token is generated by Backblaze B2's `getDownloadAuthorization` method and is valid only for the specified duration and file path.

### Expiration
All download URLs expire after 1 hour, after which any attempt to access the URL will result in a 401 Unauthorized response from Backblaze B2. This short expiration period minimizes the risk of URL sharing or unauthorized access.

## Response Handling
The document download endpoint handles responses through HTTP redirection to the generated signed URL.

### Successful Response
On successful validation and URL generation, the endpoint returns an HTTP 307 Temporary Redirect:

```typescript
return NextResponse.redirect(downloadResult.downloadUrl);
```

This redirects the client to the time-limited signed URL hosted by Backblaze B2, which then handles the actual file delivery with appropriate response headers for file disposition and content type.

### Response Headers
The final response from Backblaze B2 includes the following headers:
- **Content-Type**: Based on the document's MIME type (e.g., application/pdf)
- **Content-Disposition**: attachment; filename="{originalFilename}"
- **Content-Length**: Size of the file in bytes
- **Cache-Control**: no-cache, no-store

These headers ensure the browser treats the response as a file download rather than attempting to display the content inline.

## Error Scenarios
The endpoint handles various error scenarios with appropriate HTTP status codes and error messages.

### Invalid Parameters
When the lead ID or document ID parameters are not valid integers:

```json
{
  "error": "Invalid lead ID or document ID"
}
```
**Status**: 400 Bad Request

### Unauthorized Access
When no valid session is present:

```json
{
  "error": "Unauthorized"
}
```
**Status**: 401 Unauthorized

### Document Not Found
When no document exists with the specified documentId that belongs to the specified lead:

```json
{
  "error": "Document not found"
}
```
**Status**: 404 Not Found

### Internal Server Error
When an unexpected error occurs during processing:

```json
{
  "error": "Internal server error"
}
```
**Status**: 500 Internal Server Error

### Failed URL Generation
If the system fails to generate a download URL (e.g., due to Backblaze B2 connectivity issues), the error is logged and rethrown, resulting in a 500 Internal Server Error response.

## Data Model
The document download functionality relies on the Document model defined in the Prisma schema.

### Document Model Structure
```prisma
model Document {
  id               Int      @id @default(autoincrement())
  leadId           Int      @map("lead_id")
  filename         String
  originalFilename String   @map("original_filename")
  fileSize         Int      @map("file_size")
  mimeType         String   @map("mime_type")
  b2FileId         String   @map("b2_file_id")
  b2BucketName     String   @map("b2_bucket_name")
  uploadedBy       Int?     @map("uploaded_by")
  uploadedAt       DateTime @default(now()) @map("uploaded_at")

  // Relations
  lead Lead @relation(fields: [leadId], references: [id], onDelete: Cascade)
  user User? @relation(fields: [uploadedBy], references: [id])

  @@map("documents")
}
```

### Key Fields for Download Functionality
- **b2FileId**: The Backblaze B2 file identifier used for generating download authorizations
- **filename**: The stored filename used as the prefix in the download authorization
- **originalFilename**: The original filename preserved for user display
- **mimeType**: Used by Backblaze B2 to set the Content-Type header during download
- **leadId**: Establishes the relationship with the lead for authorization purposes

The model enforces referential integrity through the relation to the Lead model with CASCADE deletion.

## Audit Logging and Compliance
The document download endpoint implements logging for security monitoring and compliance purposes, though it does not maintain a formal audit log of successful downloads.

### Error Logging
The endpoint includes comprehensive error logging for troubleshooting and security monitoring:

```typescript
console.error("❌ DOWNLOAD: Failed to generate download URL", {
  documentId: docId,
  filename: document.filename,
  error: downloadError instanceof Error ? downloadError.message : "Unknown error",
  stack: downloadError instanceof Error ? downloadError.stack : undefined,
});
```

This logs detailed information when URL generation fails, including the document ID, filename, and error details.

### External Service Logging
The FileUploadService logs interactions with Backblaze B2:

```typescript
logger.error("Failed to generate download URL", {
  fileId,
  fileName,
  error: error instanceof Error ? error.message : "Unknown error",
});
```

### Current Limitations
The system currently lacks:
- **Successful download tracking**: There is no explicit logging of successful document downloads
- **Audit trail**: No record is maintained of which user downloaded which document and when
- **Compliance logging**: No dedicated audit log entry for document access

The application does have a general logging framework with a `fileOperation` method in the logger, but this method is not currently utilized in the download endpoint.

For compliance purposes, implementing download tracking would require adding a log entry using the logger service when a download is initiated successfully.
